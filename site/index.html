Why not the Kafka protocol?

It's the first question everyone asks. The Kafka wire protocol carries deep architectural baggage that can't be papered over:

Kafka protocol Fluorite
Schema evolution No native semantics. Bolted-on registry, burden of schema maintenance falls entirely on the end user Built into the broker. Renames, deletions, and type widening tracked in code
Partitions Hardwired into the protocol. Causes hotspots, prone to under-provisioning, and adding partitions silently drops data when consumers don't pick up the new partition—a common production failure No partitions. Broker-side offset-range dispatch handles all consumers
Storage Local disks with 3× replication. EBS alone costs ~$14,400/mo at 100K events/sec S3 with no replication overhead. ~$1,580/mo for the same workload
Durability Depends on replication factor, ISR config, and min.insync.replicas. Unclean leader election loses acked writes S3 provides 99.999999999% durability out of the box
Warehouse integration Separate topic and table abstractions. Keeping them in sync requires Kafka Connect, Debezium, or custom glue—each a source of painful inconsistencies One system. Data lands in Iceberg tables within seconds, schema evolution carries through automatically

Adopting the Kafka protocol means inheriting all of these problems. Fluorite starts fresh with a protocol designed for disaggregated storage, built-in schemas, and zero partitions.

Schema as code

Define schemas as native types in your language. No .proto files, no Avro IDL, no build plugins, no code generation step.

Fluorite auto-infers Avro schemas from your types and handles evolution—renames, deletions, type widening—tracked explicitly in your code.

@FluoriteSchema( namespace = "com.example", renames = {@Rename(from = "old_name", to = "newName")}, deletions = {"removedField"} ) public class OrderEvent { @NonNull public String orderId; public long amount; public int newName; // renamed from old_name public List<String> tags; }
@schema( namespace="com.example", renames={"old_name": "new_name"}, deletions=["removed_field"], ) @dataclass class OrderEvent: order_id: Annotated[str, NonNull] amount: int new_name: int # renamed from old_name tags: list[str]

Iceberg ingestion, built in

Data appears in Iceberg tables within seconds of being written—not minutes, not hours.

The broker's flush loop pushes records directly into per-table Iceberg buffers (hot path). Schema evolution carries through automatically—rename a field in your code and it's renamed in Iceberg.

A catch-up path runs in the background, finding any un-ingested batches via anti-join and re-reading them from S3. No data loss, no manual intervention.

No Debezium. No Kafka Connect. No Iceberg connector configs. It just works.

Run anywhere

Fluorite supports S3, GCS, and ABFS. Run on AWS, GCP, or Azure—or all three.

The broker is the same stateless binary everywhere. Bring your own object store, point the broker at it, and go.

No cloud lock-in. No vendor-specific APIs. Disaggregated storage means you own your data.
